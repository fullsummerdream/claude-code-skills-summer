# 精简专项规则

> **用途**：文风审核第 4 步（精简度审计）的执行细则。de-ai 管"像不像 AI"，本文件管"废话多不多"。
>
> **分工**：de-ai-tics-zh.md 不做字数压缩——它改的是句式骨架和表达套路；本文件专门补"提密度、删字数"这一块。两份文件在 audit-checklist.md 第 2 步和第 4 步分别被调用。

---

## 总原则

- **精简 ≠ 砍信息，是提密度。** 删的是外壳，不是内容。判断标准：删掉这句/这个词，读者少知道的**事实**为零 → 可删；少了哪怕一个事实 → 不能删，只能改写。
- **衔接 de-ai：压缩句式骨架 ≠ 改掉 LLM 套路。** de-ai 案例 6 的教训：把"越往右，X 越被组织起来……最后是工作流"压成"用法从一句话到一整套"，字数少了一半，AI 味一点没少——因为"从 X 到 Y"的骨架还在。判断一句话像不像 LLM，**看骨架不看字数**。
- **顺序：先去 AI 味，再精简。** 先按 de-ai-tics-zh.md 改掉套路句（第 2 步），再按本文件提密度（第 4 步）。反着做会出事：把 AI 套路句压缩得更短，等于把问题藏得更深。
- **精简不碰两类东西**：① 读者第一次见到的关键定义；② 论证链条上缺了就断的环节。这两类哪怕写得啰嗦，也只能改写不能删。

---

## 一、删冗余

### 1.1 同义重复

**定义**：同一句内或相邻两句，同一信息说了两遍。常见形态：前半句说了，后半句换个说法再说一遍；或者上一句说了，下一句开头又复述一遍。

**识别方法**：
- 把每句话压成一个信息点，写下来。相邻两个信息点相同 → 有一处是重复。
- 重点盯"也就是说 / 换句话说 / 换言之 / 这意味着"——后面跟的 90% 是复读。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| 这个方案可以显著降低成本，能让公司的开支大幅减少。 | 这个方案可以显著降低成本。 | 后半句是前半句的同义复读 |
| 会议改到下午三点。也就是说，原定上午的会议推迟到下午三点。 | 会议改到下午三点。 | "也就是说"引导的整句复读 |
| 他做事非常认真，对每个细节都一丝不苟、非常认真。 | 他做事认真，对每个细节一丝不苟。 | 句尾重复的"非常认真" |

### 1.2 同义堆叠

**定义**：三个（或更多）近义形容词/副词排比堆砌，看着有气势，实际信息等于其中一个词。和 de-ai 1.5"可 X、可 Y、可 Z 三连"同源，但 de-ai 管的是"AI 腔"，这里管的是"字数"——即便不是 AI 写的，三件套也该砍。

**识别方法**：
- 顿号连接的三个及以上近义词 → 触发警报。
- 逐个盖住测试：盖住其中两个词，句子意思变不变？不变 → 留最准的 1 个，其余删。
- 留哪个：留**最具体、最有区分度**的那个，不是留最响亮的那个。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| 我们需要一个快速、高效、敏捷的响应机制。 | 我们需要一个快速的响应机制。 | 三个近义词留 1 个（"快速"最具体，说速度） |
| 这份报告全面、系统、深入地分析了市场趋势。 | 这份报告深入分析了市场趋势。 | "全面、系统"是修饰充气，"深入"有区分度 |
| 流程要透明、公开、阳光化。 | 流程要透明。 | "公开、阳光化"与"透明"同义 |

### 1.3 修饰冗余

**定义**：副词/形容词堆砌，删掉不影响句子传达的事实。高危词：非常、十分、极其、相当、很、非常之、高度（作副词）、进一步（无具体所指时）、积极（作副词）、有力、有效（无对比时）。

**识别方法**：
- 逐词删除测试：把这个修饰词删掉，句子的事实含量少不少？不少 → 删。
- 警惕"程度副词 + 已经表程度的词"双重强化："非常巨大""极其重要""十分关键"——"巨大/重要/关键"本身已带程度。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| 这是一个非常重要的问题，我们必须高度重视。 | 这个问题必须优先处理。 | "非常重要""高度"两层程度副词，换成动作 |
| 系统运行得非常稳定，极少出现故障的情况。 | 系统一年故障两次。 | 两个修饰词都不如一个具体数字（见第四节） |
| 我们取得了非常显著、十分突出的成绩。 | 用户留存率从 40% 提到 65%。 | "非常显著、十分突出"双堆 + 抽象名词，换成事实 |

## 二、合并

### 2.1 短句合并

**定义**：相邻两个短句共享主语或宾语，拆开写是浪费——主语说两遍、句号多一个、读者多停一次。

**识别方法**：
- 看相邻两句的主语。相同 → 多半可合。
- 看第二句开头是不是"它 / 这 / 该 X / 同时"——回指上一句 → 可合。
- 合并上限：合完一句话不超过两层意思。塞三层以上宁可不合。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| 这个工具支持批量导入。它还支持自动去重。 | 这个工具支持批量导入和自动去重。 | 重复的主语和一个句号 |
| 新版接口响应更快。新版接口的错误提示也更清楚。 | 新版接口响应更快，错误提示也更清楚。 | 重复的主语"新版接口" |
| 数据存在本地。数据不会上传到云端。 | 数据存在本地，不上传云端。 | 重复主语 + "不会……的情况"式赘余 |

### 2.2 段落合并

**定义**：相邻两段讲的是同一件事的不同侧面，或者第二段只是第一段的展开解释——分两段是排版充气，合一段更紧。

**识别方法**：
- 给每段写一句话梗概。两段梗概是"总—分"或"重述"关系 → 合。
- 第二段开头是"具体来说 / 例如 / 换句话说"且只有一两句 → 并进上一段。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| （段1）团队本季度的重点是稳定性。（段2）具体来说，我们要把崩溃率降到 0.1% 以下，并把平均响应时间控制在 200ms 内。 | 团队本季度的重点是稳定性：崩溃率降到 0.1% 以下，平均响应时间控制在 200ms 内。 | 段落分隔 + "具体来说"过渡 |
| （段1）老系统有三个致命问题。（段2）这三个问题分别是：单点故障、无法水平扩展、日志缺失。 | 老系统有三个致命问题：单点故障、无法水平扩展、日志缺失。 | "这三个问题分别是"的二次引导 |
| （段1）我们决定放弃自研。（段2）原因是自研成本高、周期长，而市面方案已经够用。 | 我们决定放弃自研：成本高、周期长，市面方案已经够用。 | 段间分隔 + "原因是"连接词 |

### 2.3 列表合并

**定义**：列表项之间可以归类——好几条其实在说同一类事，逐条平铺显得长，归组后结构更清楚、字数更少。

**识别方法**：
- 给每条列表项打一个类别标签。同一标签出现 2 次以上 → 归组。
- 列表超过 7 条还没有任何分组 → 大概率该归组（人脑一次处理不了 7+ 平铺项）。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| - 修复登录页崩溃<br>- 优化首页加载速度<br>- 修复支付页报错<br>- 压缩图片体积<br>- 修复个人中心白屏 | **修复**：登录页崩溃、支付页报错、个人中心白屏<br>**性能**：首页加载提速、图片压缩 | 五条平铺归成两组，每类一个引导词 |
| - 张三负责前端<br>- 李四负责后端<br>- 王五负责测试<br>- 张三兼任发版 | **前端/发版**：张三；**后端**：李四；**测试**：王五 | 四条同构句的重复句式"负责" |
| - 支持 CSV 导入<br>- 支持 Excel 导入<br>- 支持 JSON 导入 | 支持 CSV / Excel / JSON 导入 | 三次重复的"支持……导入"句式 |

## 三、去垫话

> 本节与 de-ai 的"元叙述 / 导游词"（案例 10、14、19、21、22）同族，但视角不同：de-ai 管的是"这类句子很 AI"，这里管的是"这类句子占字数"。一个句子可能不 AI 但仍是垫话（真人写公文也堆"综上所述"），两种文件都要过。

### 3.1 开场白

**定义**：正文开始前的一段"套近乎/拉背景"，删掉后文章从第二段读起完全成立。

**识别方法**：
- 高危词表：众所周知、随着……的发展、在当今社会、在……的大背景下、近年来、不得不承认、毋庸置疑。
- 测试：从第二段开始读。成立 → 第一段是垫话。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| 随着移动互联网的飞速发展，短视频已经成为人们生活中不可或缺的一部分。在这样的背景下，我们调研了用户的观看习惯。 | 我们调研了 1200 名用户的短视频观看习惯。 | 两句时代背景垫话，顺手补上具体样本量 |
| 众所周知，数据是新时代的石油。因此数据治理至关重要。 | 数据治理要解决三个问题：（直接进内容） | "众所周知"开头的断言式开场 |
| 在当今竞争激烈的市场环境下，企业要想生存，就必须重视效率。下面谈谈我们的做法。 | （整段删，直接谈做法） | 开场白 + 过渡语双重垫话 |

### 3.2 过渡语

**定义**：段落/章节之间宣告"接下来我要讲什么"的句子。删掉后，靠标题和段落顺序读者照样跟得上。

**识别方法**：
- 高危词表：接下来我们来看、下面介绍一下、说到这里、我们再来看看、讲完了 X，我们再讲 Y、首先……其次……最后（作纯结构信号时）。
- 删除测试（de-ai 三测试第一条）：删掉这句话，前后段落接得上 → 垫话。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| 介绍完技术方案，接下来我们看一下实施计划。实施计划分三个阶段…… | 实施计划分三个阶段…… | 过渡句整句删，章节标题已定位 |
| 下面介绍一下测试结果。测试覆盖了 500 个用例，通过率 98%。 | 测试覆盖 500 个用例，通过率 98%。 | "下面介绍一下"五字垫话 |
| 讲完了问题，我们再来看看方案。方案的核心是缓存分层。 | 方案的核心是缓存分层。 | "讲完了 X 再讲 Y"的路标句 |

### 3.3 收尾套话

**定义**：文章/段落末尾不增加信息的总结性客套。读者读完最后一段，自己知道文章结束了，不需要被告知。

**识别方法**：
- 高危词表：综上所述、总而言之、总的来说、由此可见、希望对大家有帮助、欢迎批评指正（非必要场合）、让我们拭目以待、未来可期。
- 测试：删到最后一段只剩事实和结论。如果剩的那句只是"把上文再说一遍"或"客套"，删。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| 综上所述，缓存分层方案在成本、性能、可维护性上都优于现有方案，值得推广。 | （末句前已是三方面的数据对比，整句删） | "综上所述"引导的复读式总结 |
| 以上就是本次分享的全部内容，希望对大家有帮助，谢谢大家。 | （整句删，或只留"提问环节"） | 客套三连 |
| 总而言之，这个项目虽然遇到了一些困难，但我们最终还是克服了，取得了不错的成绩。 | 项目延期两周上线，崩溃率从 2% 降到 0.3%。 | 情绪化总结换成具体结果 |

### 3.4 自我指涉

**定义**：文章指自己——"本文将讨论""上文提到""如前所述""见下文"。读者正在读、刚读过、马上会读到，都不需要被提醒文章的存在。

**识别方法**：
- 搜索词：本文、本报告、本节、上文、前文、如下、见下、将在第 X 节。
- 区分真假：指**别处文档**（"见附录 B""详见运维手册"）是导航，保留；指**自身相邻段落**的是垫话，删。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| 本文将讨论三个问题：性能、成本、安全。 | 三个问题：性能、成本、安全。 | "本文将讨论"五字自我指涉 |
| 上文提到了缓存方案，下面我们详细展开。缓存分三层…… | 缓存分三层…… | "上文提到 + 下面展开"双重垫话 |
| 测试结果如下表所示。 | （删，表格紧跟着，读者看得见） | "如下表所示"引导句 |

## 四、信息密度

### 4.1 每句必须有新信息

**规则**：读完一句，问自己"这句给了什么前面没有的信息？"答不上来 → 删，或与邻句合并。

**操作**：逐句做"信息点登记"——每句压成一个短语写下来。登记完检查：
- 两个登记项相同 → 同义重复（回 1.1）。
- 某个登记项是"空"（"这很重要""不容忽视""值得关注"）→ 整句删。

| 原文 | 判定 |
|---|---|
| 安全性非常重要，绝不能忽视安全问题。 | 两句都只登记出"安全重要"，且是空话 → 合成一句"安全是硬指标：泄露一次客户数据就解约"或直接删 |
| 这个决定不容易。我们反复讨论了很久。 | "不容易"和"反复讨论"登记为同一信息 → 留"这个决定讨论了六轮"（顺手具体化） |

### 4.2 抽象 → 具体：同字数信息更多

**规则**：空泛评价换成可核验的事实，字数不变甚至更少，信息量翻倍。这是"提密度"的主战场——密度低往往不是句子多，而是句子空。

**识别方法**：
- 高危空泛词：效果很好、反响热烈、意义重大、显著提升、广受好评、反响强烈、成果丰硕、反响不错。
- 追问测试：好到什么程度？谁说的？有数字吗？答得出 → 把答案写进去；答不出 → 删（作者自己都不知道，留着是充气）。

| 原文 | 精简后 | 删了什么 |
|---|---|---|
| 优化后系统性能有了显著提升。 | 响应时间从 3s 降到 200ms。 | "显著提升"空话 → 具体数字 |
| 本次活动取得了圆满成功，受到广泛好评。 | 活动到场 300 人，回收满意度问卷 4.6/5。 | "圆满成功、广泛好评" → 两个可核验数 |
| 新流程大大提高了工作效率。 | 审批从 3 天缩到 4 小时。 | "大大""效率"双空泛 → 时间对比 |

### 4.3 密度检查（定量）

**方法**：通读全文后，凭记忆复述出信息点（事实、数据、结论、动作），数出个数 N；再数全文总句数 M。

```
密度比 = N / M     目标 > 0.8
```

- 密度比 ≤ 0.5：一半句子是空转，需要结构性重写，不是逐句删能救的。
- 0.5 < 密度比 ≤ 0.8：逐句过本文件第一~三节。
- 密度比 > 0.8：达标，转入微调。
- 注意：复述不出来的句子才算零信息句——读者记不住，等于没写。

逐句扫，命中任意一条就动手：

- [ ] 有没有"也就是说 / 换句话说 / 换言之"引导的复读句？

- [ ] 相邻两句主语相同？合了没有？
- [ ] 相邻两段是"总—分 / 重述"关系？合了没有？
- [ ] 列表超过 7 条还没分组？能归类的归组没有？
- [ ] 开头有没有"众所周知 / 随着……的发展 / 在当今社会"？
- [ ] 段间有没有"接下来我们来看 / 下面介绍一下 / 讲完了 X 再讲 Y"？
- [ ] 结尾有没有"综上所述 / 总而言之 / 希望对大家有帮助"？
- [ ] 有没有"本文将 / 上文提到 / 如下表所示"这类自我指涉？
- [ ] 逐句登记信息点，有没有登记为"空"的句子（很重要/不容忽视/值得关注）？
- [ ] "效果很好 / 显著提升 / 广受好评"换成数字或事实没有？换不出 → 删。
- [ ] 每个"我 / 我们"删掉后句子还成立的，删了没有？（与 de-ai 案例 16 共用诊断）
- [ ] 密度比算了没有？> 0.8 才算过。
- [ ] 最后一遍：删完的字数有没有汇报？压缩比多少？（见第六节）

## 六、精简报告格式

精简完成后输出三行统计 + 一张分类表：

```
原字数：XXXX
精简后字数：XXXX
压缩比：XX.X%（删掉的字数 / 原字数）
```

| 类别 | 删改处数 | 占总删减字数比 | 典型例子（一处即可） |
|---|---|---|---|
| 冗余（同义重复/堆叠/修饰） | X | XX% | "非常显著、十分突出" → 数字 |
| 合并（短句/段落/列表） | X | XX% | 两句同主语合一 |
| 垫话（开场/过渡/收尾/自指） | X | XX% | "综上所述……"整句删 |

**判定参考**：
- 一般 LLM 初稿：压缩比 20%–40% 属正常区间；低于 10% 要怀疑没认真扫，高于 50% 要怀疑误删了信息。
- 压缩比高 ≠ 好。报告里若"垫话"占比最大，说明原稿骨架还行只是水多；若"冗余"占比最大，说明原稿句子级问题重，建议作者回看第一节。
- 报告**必须**附"误删自查"一句：抽查 3 处最长的删除，确认没有删掉事实。有 → 恢复并注明。
